Pour commencer

Nmap est un logiciel gratuit, open-source pour scanner les ports. Il est disponible 
pour UNIX et Windows. Il dispose d'une interface graphique optionnelle, ZenNmap, et 
prend en charge une grande variété de types de balayage, chacune avec des avantages 
et des inconvénients. Ce tutoriel décrit certains types d'analyse faite par NMAP ainsi 
qu'expliquer leurs avantages et comment ils fonctionnent réellement. J'offre également 
des conseils sur les types de scan qui serait mieux de faire contre tels et tels types 
d'hôte. Je suppose que vous avez installé Nmap (et que vous savez comment l'installer. 
Les instructions sont disponibles sur le site Web Nmap, https://nmap.org/), et que vous 
avez les privilèges requis pour exécuter les scans détaillés (de nombreux scans nécessitent 
des privilèges root ou Administrateur).

Exemple

1. Scan de base [-sT, -sS]
Les deux types d'analyse de base les plus utilisés dans Nmap sont « TCP connect scan » [ -sT ] 
et le scan SYN (également connu sous le nom de « stealth scanning » ) [ - sS ] . Ces deux types 
de scans sont expliqués en détail ci-dessous .

1.1 TCP connect scan [-sT]
Ce scan permet un scan de base des ports , et il tentera de se connecter à chaque ports. 
Une fois l'analyse terminée , les ports pour lesquels une connexion peut être établie sont 
répertoriés comme open, on dit que le reste est closed. Cette méthode d'analyse est très efficace , 
et fournit une image claire des ports que vous pouvez et ne pouvez pas accéder . Mais ! 
Parce qu'il y a toujours un mais. C'est très facile de vous détecter. Si un système de détection 
d'intrusion ou un pare-feu ou est en cours d'exécution sur la machine de votre victime et que 
vous tentez de se connecter à chaque port , le système saura ce qui se passe et fermera tout ou avertira l'administrateur.

 
1.2 SYN stealth scan [-sS]
Pour établir une connexion TCP « furtive », l'attaquant doit envoyer un paquet SYN à son destinataire , 
qui répondra alors avec un SYN lui aussi, et un ACK , pour affirmer qu'il a bien recu notre SYN. 
L'attaquant envoie ensuite lui aussi un paquet ACK pour affirmer qu'il a bien recu le paquet ACK de la victime. 
Tout ce joue ici, si les paquets SYN / ACK ont été envoyé et recu des deux coté. Alors le port est ouvert et 
la connexion TCP peut être établie.
C'est beaucoup mieux qu'envoyer des requête TCP un peu partout au hasard.

2. FIN, Null et « Xmas Tree Scans » [-sF, -sN, -sX]
Avec la multitude de pare-feu modernes qui recherche maintenant les scans SYN , ces trois types d'analyse peuvent 
être utiles à divers degrés. Chaque type de scan fait référence aux drapeaux définis dans l'en-tête TCP. L'idée derrière ces types 
d' analyses est qu'un port fermé doit répondre obligatoirement par un paquet RST, alors qu'un port ouvert devrait juste laisser tomber 
votre requête ( il cherche plutôt les paquets SYN ).

 
Le scan FIN envoie un paquet avec seulement le flag FIN, le scan Xmas Tree définit les flags FIN, URG et PUSH 
(voir un bon livre TCP / IP pour plus de détails ) et le scan Null envoie un paquet sans aucun flag. Ces types de scan 
travailleront contre tous les systèmes où le protocole TCP / IP suit le code RFC 793. Microsoft Windows ne suit pas la RFC , 
il va ignorer ces paquets même sur les ports fermés. En exécutant une requête SYN avec l'un de ces scans, il sera possible de détecter 
si le système est un Windows ou non. Si le paquet SYN montre des ports ouverts , et que les paquets FIN / NULL / XMAS non, 
vous êtes alors très probablement sur un système Windows.

4. Scan UDP [-sU]
Un problème majeur avec cette technique est que, quand un pare-feu bloque les connexions sortantes, 
vous aurez un message d'erreur mais le port vous apparaîtra ouvert. Ce qui n'est pas toujours le cas. 
On appel cela des « false positives ». Un autre inconvénient avec le scan UDP est la vitesse à laquelle il peut être exécutée. 
La plupart des systèmes d'exploitation limitent le nombre de requête qui peuvent être générés en un certain laps de temps, 
ce qui ralentit de beaucoup la vitesse du scan. Nmap ajuste sa vitesse de scan en conséquence pour éviter d'inonder un réseau avec des paquets inutiles. 
Un point intéressant à noter ici est que Microsoft ne limitent pas la fréquence de génération d'erreur de Port Inaccessible, et il est donc facile pour 
numériser 65.535 Ports UDP d'une machine Windows en très peu de temps ! Il est généralement pas utile pour la 
plupart des types d'attaque d'utiliser le scan UDP, mais il peut révéler informations sur les services ou les trojans 
qui reposent sur ce protocole, par exemple, SNMP , NFS, le cheval de Troie et de nombreux autres services.

5. Scan de Protocole IP [-sO]
La tentative des scans de protocole IP sert à déterminer si les protocoles IP sont supportés sur la cible. 
Nmap envoie un paquet IP brut sans en-tête de protocole additionnel, à chaque protocole sur la machine cible. 
La réception d'un message nous dit si le protocole n'est pas en cours d'utilisation, sinon le port est ouvert. 
Je crois que seul Windows fera un rapport de tous les protocoles ouverts lors de la demande. Mais je ne suis pas certain.

 
6. Scan « Idle » [-sI]

Le scan Idle est une technique très avancée où aucun paquet ne sont
envoyée à la cible. Un hôte zombie (et le port) doivent être spécifiés pour ce type d'analyse. 
L'hôte zombie doit satisfaire certains critères essentiels pour le bon fonctionnement de cette analyse. 
Ce type de balayage fonctionne en exploitant la séquence « predictable IP fragmentation ID » sur l'hôte zombie 
afin de déterminer les ports ouverts ou non. L'analyse vérifie le IPID du zombie ,puis demande une connexion à la machine cible, 
ce qui semble à être une connexion 127.0.0.1 depuis la machine attaqué. Si le port cible est ouvert, une session de SYN / ACK commence 
et est envoyé à partir de la machine zombie. Grâce à cette technique, il est possible d'analyser chaque port de la cible comme 
si c'était la machine cible (le zombie) qui faisait le scan. Bien sûr, les tentatives de connexion seront probablement enregistrées , 
de sorte que le système et ses logs contiennent l'adresse IP de l'attaquant. En outre, certains IDS et pare-feu font des tentatives 
pour détecter les paquets de ce genre sur le réseau. Tant que l'hôte zombie et l'attaquant sont tous les deux " sur Internet" ou sur 
le même réseau, les techniques pour identifier les paquets ne fonctionneront pas. Le plus dur qui est à faire est de créer un zombie.
